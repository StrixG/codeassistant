# File Agent (День 34) — целеориентированный агент по файлам проекта

Агент, которому ставят **цель** («обнови документацию», «добавь лицензионные
заголовки»), а не пошаговые команды. Он сам решает, какие инструменты и в каком
порядке вызывать: ищет, читает, анализирует, создаёт и правит файлы целевого
репозитория через function calling DeepSeek.

Пакет переиспользует существующую инфраструктуру и добавляет поверх неё только
файловые инструменты и агентный цикл:

| Переиспользовано | Откуда |
|---|---|
| Клиент DeepSeek (function calling, retry, учёт токенов) | `assistant.core.llm.DeepSeekClient` |
| Реестр / исполнитель инструментов | `assistant.core.tools.Tool`, `ToolRegistry`, `ToolExecutor` |
| Семантический поиск по Chroma-индексу кода | `assistant.core.rag.RagSearcher` |
| Конфиг из `.env` | `assistant.config.Config` |

## Архитектура

```
python -m file_agent "цель" [--dry-run]
        │
        ▼
 file_agent/agent.py  ──►  цикл (до 15 итераций):
        │                    DeepSeek(chat, tools) ──► tool_calls?
        │                        ├─ да  → выполнить инструменты, вернуть результаты в контекст, повтор
        │                        └─ нет → финальный ответ, стоп
        ▼
 file_agent/tools.py  ──►  5 инструментов над TARGET_REPO_PATH:
     list_files · read_file · search(text|semantic) · write_file · edit_file
```

Каждый шаг печатается в консоль (какой инструмент, аргументы, краткий результат)
— это трасса для демо-видео.

### Инструменты (`file_agent/tools.py`)

| Инструмент | Назначение |
|---|---|
| `list_files(glob_pattern)` | Список файлов по маске (`vector-config/**/*.kt`). |
| `read_file(path, offset?, limit?)` | Чтение текстового файла (с ограничением размера; опц. срез по строкам). |
| `search(query, mode)` | `text` — grep по трекнутым файлам (`git grep`); `semantic` — ближайшие чанки из Chroma-индекса кода. |
| `write_file(path, content)` | Создание/перезапись файла. |
| `edit_file(path, old_text, new_text)` | Точечная замена; `old_text` должен встречаться **ровно один раз**, иначе правка отклоняется. |

У каждого инструмента — докстринг и JSON-схема параметров для function calling,
из которых модель понимает, когда его вызывать.

### Безопасность

- Каждый путь резолвится (`Path.resolve()`) и проверяется, что он **внутри
  `TARGET_REPO_PATH`**; выход за корень (`../`, абсолютные пути, симлинки) —
  ошибка `FileToolError`.
- Служебные пути `.git/**` не читаются и не пишутся.
- Бинарные файлы (NUL-байт в начале) не читаются и не переписываются как текст.
- Репозиторий захватывается в конструкторе `FileTools`; модель может передать
  путь, но не может перенаправить инструменты на другой каталог.

### Лимиты

| Что | Значение | Где |
|---|---|---|
| Итераций агентного цикла | 15 | `agent.MAX_ITERATIONS` |
| Размер чтения файла | 100 000 байт | `tools._MAX_READ_BYTES` |
| Строк в `list_files` | 400 | `tools._MAX_LIST` |
| Строк в текстовом поиске | 200 | `tools._MAX_GREP_LINES` |
| Хитов семантического поиска | 8 | `tools._MAX_SEMANTIC_HITS` |

При исчерпании лимита итераций цикл останавливается, `hit_limit=True`, код
возврата CLI — `2`.

### Режим `--dry-run`

`write_file` / `edit_file` **не пишут на диск**: изменения копятся в оверлее в
памяти. `read_file` учитывает оверлей, поэтому цепочка правок остаётся
согласованной. В конце выводится единый **unified diff** всех накопленных
изменений.

## Переменные окружения

Читаются из `.env` (см. `.env.example`):

| Переменная | Назначение |
|---|---|
| `DEEPSEEK_API_KEY` | Ключ DeepSeek (единственный источник ключа). |
| `TARGET_REPO_PATH` | Корень целевого репозитория (например, `element-android`). |
| `CHROMA_PATH` | Путь к Chroma-индексу (для `search mode=semantic`). |

## Запуск

```bash
# читающий сценарий
python -m file_agent "Найди все места использования класса AnalyticsTracker и создай в корне usage_report.md с таблицей файл/строка/как используется и выводом об архитектуре"

# пишущий сценарий: сначала предпросмотр, потом реально
python -m file_agent "Добавь лицензионный заголовок во все .kt-файлы vector-config, где его нет; выведи список исправленных" --dry-run
python -m file_agent "Добавь лицензионный заголовок во все .kt-файлы vector-config, где его нет; выведи список исправленных"
```

> Для `search mode=semantic` нужен построенный индекс кода:
> `python -m assistant index` (индексатор кладёт Kotlin-чанки с `source=code`).
> Без него семантический режим корректно вернёт подсказку, а текстовый поиск
> (`git grep`) работает всегда.

## Тесты

```bash
.venv/bin/pytest tests/test_file_agent_tools.py tests/test_file_agent_loop.py -q
```

Покрывают: валидацию путей (выход за корень → ошибка), отказ по `.git`/бинарным
файлам, `edit_file` при отсутствующем и неоднозначном `old_text`, `--dry-run` не
пишет на диск и собирает diff, остановку цикла по лимиту итераций и чистую
остановку по ответу без tool calls. LLM в тестах замокан.
